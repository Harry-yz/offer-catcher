# Offer Catcher — AI求职智能匹配系统方案说明书

---

## 一、项目背景

### 1.1 为什么做这个项目

每年秋招季，身边同学都在经历同样的痛苦：打开招聘平台，几百个岗位刷下来，真正适合自己的可能就那么几个，但要花好几天才能筛选出来。更头疼的是，简历投出去石沉大海，却不知道问题出在哪。

我观察到两个核心问题：

**岗位筛选靠运气。** 大多数同学的做法是海投——看到"Java开发""产品经理"就投，根本不看自己和岗位到底有多匹配。结果是简历通过率低，面试机会少，时间全浪费在等待上。

**简历优化靠感觉。** 很多人写简历就是把自己做过的事罗列一遍，不会针对目标岗位去调整表述。比如同样是"负责后端开发"，换成"主导后端架构设计，支撑日均100万请求"，通过率完全不一样。

### 1.2 现有方案的局限

| 方案 | 能做什么 | 做不好什么 |
|------|----------|------------|
| 招聘平台筛选 | 按关键词过滤 | 无法理解语义，"会Python"和"精通Python"分不清 |
| 找人内推 | 了解岗位真实需求 | 成本高，不是每个人都有内推资源 |
| 问ChatGPT | 能分析简历 | 缺乏求职场景的专业知识，回答泛泛 |

所以我想做一个工具：**上传简历和JD，就能知道匹配度，知道怎么改简历，知道面试会问什么。**

## 二、整体设计

### 2.1 产品定位

Offer Catcher 不是又一个招聘平台，而是一个**求职辅助工具**。用户上传自己的简历和目标岗位的JD，系统自动完成三件事：

1. **告诉你匹不匹配** —— 量化评分，精确到每个技能点
2. **告诉你怎么改** —— 简历优化建议，具体到某句话怎么调整
3. **帮你准备面试** —— 生成面试题库，模拟面试对话

### 2.2 三个核心功能

**功能一：智能岗位匹配**
上传简历和多个岗位JD，系统按匹配度排序，列出每个岗位的匹配项和不匹配项，给出简历优化建议。

<!-- 截图：岗位匹配结果页面 -->
<!-- ![岗位匹配结果](./images/match-results.png) -->

**功能二：智能面试准备**
上传简历和一个目标岗位JD，系统生成针对性的面试题库、自我介绍，还能模拟面试对话并实时评分。

<!-- 截图：面试准备页面 -->
<!-- ![面试准备](./images/interview-prep.png) -->

**功能三：候选人深度验证**
面试官模式，上传候选人简历和岗位JD，系统设计针对性面试问题，帮助识别简历中需要重点验证的经历。

### 2.3 系统架构

整体采用分层架构，前端和后端分离，核心逻辑都在后端：

```
┌─────────────────────────────────────────────┐
│               用户界面层                      │
│        HTML/CSS/JS + 渐变背景 + 毛玻璃卡片     │
├─────────────────────────────────────────────┤
│               业务逻辑层                      │
│     岗位匹配 │ 面试准备 │ 候选人验证           │
├─────────────────────────────────────────────┤
│               AI服务层                       │
│     MinerU(文档解析) + MiMo(大模型推理)        │
├─────────────────────────────────────────────┤
│               基础设施层                      │
│     缓存管理 │ 配置管理 │ 错误处理             │
└─────────────────────────────────────────────┘
```

## 三、技术实现

### 3.1 技术选型理由

| 技术 | 为什么选它 |
|------|-----------|
| **Flask** | Python生态里最轻量的Web框架，这个项目不需要Django那种重型框架 |
| **MinerU** | 开源文档解析工具，中文PDF和图片识别效果好，比Tesseract准很多 |
| **MiMo** | 小米自研大模型，中文理解能力强，API调用成本低 |
| **原生HTML/CSS/JS** | 前端功能简单，没必要上React/Vue，反而增加复杂度 |
| **文件缓存** | 用MD5做key，相同文件不重复解析，省API调用次数 |

### 3.2 文档解析流程

简历和JD可能是PDF、图片或纯文本，格式不统一。处理流程是：

```
用户上传文件
    ↓
判断文件类型
    ├── PDF → MinerU提取文本 → MiMo结构化解析
    ├── 图片 → MinerU OCR识别 → MiMo结构化解析
    └── 文本 → 直接调用MiMo结构化解析
    ↓
返回结构化数据（姓名、技能、项目经历等）
```

解析结果会缓存，同一份文件不会重复调用MinerU和MiMo。

### 3.3 匹配算法

匹配不是简单的关键词对比，而是让AI理解语义后打分。具体来说：

**输入：** 结构化的简历数据 + 结构化的JD数据

**处理过程（多Agent协作）：**
1. **岗位匹配Agent** —— 逐个分析简历和每个JD的匹配度，输出分数和匹配/不匹配项
2. **报告生成Agent** —— 把多个匹配结果排序，生成综合分析报告

**评分维度：**
- 技能匹配（40%）：精确匹配、模糊匹配、相关技术匹配
- 经验匹配（25%）：工作年限、项目经验相关性
- 项目质量（20%）：量化成果、技术深度
- 综合素质（15%）：教育背景、证书等

### 3.4 面试准备功能

这是用户反馈最好的功能。流程是：

1. 系统根据简历和JD生成5-8道面试题
2. 每道题包含：问题、考察意图、参考答案、追问方向
3. 用户可以在"模拟面试"模式下练习
4. 用户回答后，AI给出评分和标准答案

面试题不是泛泛的"说说你的优缺点"，而是针对具体岗位和简历定制的。比如简历里写了"压降Token成本40%"，系统会追问"这个40%是怎么算出来的？基准是什么？"

### 3.5 前端设计

前端追求简洁高级感，参考了Linear、Vercel等产品的设计风格：

- **背景**：流动的弥散渐变，暗色调中融入极光绿与深海蓝
- **卡片**：高斯模糊（毛玻璃效果）+ 半透明白边框
- **交互**：悬停时卡片上移10px，图标微旋转，阴影扩散
- **进度**：思维链式显示，用户能看到每一步在做什么

全部用纯CSS实现，零JavaScript依赖（除了业务逻辑），页面加载很快。

## 四、项目亮点

### 4.1 多Agent协作

不是把所有任务塞给一个AI，而是拆分成多个专职Agent：
- 岗位匹配Agent 只负责打分
- 报告生成Agent 只负责整理输出
- 面试准备Agent 只负责生成题目

每个Agent的提示词都针对自己的任务优化过，比一个通用Agent效果好很多。

### 4.2 缓存机制

相同文件不会重复解析。简历和JD的解析结果用MD5哈希做key缓存到本地。对于求职者来说，同一份简历分析多个岗位，只有第一次需要等，后面都是秒出结果。

### 4.3 思维链式进度

不是转圈等，而是按顺序显示：
```
✅ 正在解析候选人简历...
✅ 简历解析完成
🔄 正在分析岗位 1/3: 腾讯 - AI方向...
✅ 岗位分析完成
🔄 正在生成综合报告...
✅ 匹配完成
```

用户知道系统在做什么，不会焦虑地等待。

### 4.4 错误处理

API调用可能失败（超时、限流等），系统会：
1. 自动重试（最多3次）
2. 显示具体错误原因
3. 提供解决方案提示
4. 支持一键重试

## 五、效果评估

### 5.1 功能测试

用测试简历和真实JD跑了完整流程：

| 测试项 | 结果 |
|--------|------|
| PDF简历解析 | 成功提取姓名、技能、项目经历 |
| 图片JD解析 | 成功识别公司、职位、要求 |
| 多岗位匹配 | 3个JD全部匹配成功，分数排序合理 |
| 面试题生成 | 生成5道针对性题目，追问逻辑清晰 |
| 模拟面试 | 评分和反馈具体，不是套话 |

### 5.2 性能表现

| 指标 | 首次 | 缓存后 |
|------|------|--------|
| 简历解析 | 15-30秒 | 瞬间 |
| JD解析 | 5-15秒 | 瞬间 |
| 单岗位匹配 | 5-10秒 | 5-10秒 |
| 面试题生成 | 5-10秒 | 5-10秒 |

缓存对解析阶段效果明显，匹配和面试生成因为每次内容不同，无法缓存。

### 5.3 匹配准确度

测试了几个真实场景：

| 简历 | 岗位 | 系统评分 | 主观判断 |
|------|------|----------|----------|
| Python后端开发 | Python后端开发 | 85分 | 高度匹配 |
| Python后端开发 | AI算法工程师 | 60分 | 一般匹配 |
| Python后端开发 | 前端React开发 | 40分 | 不匹配 |

评分和主观判断基本一致。

## 六、不足与改进

### 当前不足

1. **需要手动上传JD** —— 不能自动从招聘平台抓取岗位
2. **没有用户系统** —— 数据存在本地，换个电脑就没了
3. **移动端体验一般** —— 主要针对桌面端设计
4. **匹配算法依赖AI** —— 不同模型、不同prompt可能给出不同分数

### 后续计划

1. 接入招聘平台API，实现岗位自动搜索
2. 添加用户登录和历史记录
3. 优化移动端布局
4. 引入更精准的匹配算法，减少对AI的依赖

---

## 七、总结

这个项目从一个实际痛点出发，用AI技术解决了求职过程中的信息不对称问题。技术上尝试了多Agent协作、文档解析、缓存优化等方案；产品上覆盖了从岗位筛选到面试准备的完整流程。

最大的收获是学会了如何把AI能力包装成用户能用的产品——光有技术不够，还得考虑用户体验、错误处理、性能优化这些"最后一公里"的问题。
